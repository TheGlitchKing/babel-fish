# Integration with `semantic-memory`

`semantic-memory` (formerly `semantic-sidekick`) is the unified memory-layer plugin for Claude Code. It consumes babel-fish's auto-generated project map as a **`project-map` corpus** and additionally extracts the `01-vocabulary.md` file into a structured `glossary.json` side-channel that powers translation verbs.

This doc covers the consumer-side integration. For the data contract specification, see [`docs/glossary-contract.md`](./glossary-contract.md).

## How the integration works

```
┌──────────────────┐    pre-commit hook     ┌──────────────────┐
│   babel-fish     │ ───────────────────→   │  .babel-fish/    │
│ (producer)       │   regenerates 19       │  01-vocabulary.md│
│                  │   sections              │  02-...md        │
└──────────────────┘                         │  ... etc.        │
                                             └────────┬─────────┘
                                                      │
                                                      │ file-watcher detects change
                                                      ↓
┌──────────────────┐    indexes as           ┌──────────────────┐
│ semantic-memory  │ ────────────────────    │  .semantic/      │
│ (consumer)       │ ←─── reads ─────────    │  project-map/    │
│                  │     extracts            │   INDEX (HNSW)   │
│                  │                          │   glossary.json  │
└──────────────────┘                         └──────────────────┘
        │
        │ exposes 3 MCP verbs (when corpus active)
        ↓
   translate / reverse_translate / list_vocabulary
        │
        │ used by
        ↓
   subagents, slash commands, query rewriting
```

## Setup

### Prerequisites

- babel-fish installed (any 2.x version)
- semantic-memory installed (1.1.0+ for full integration; 1.0.0a only ships the contract documentation)
- Both plugins active in the same project

### Enable the project-map corpus

Edit `.semantic/corpora.json` (auto-generated by semantic-memory's smart-middle activation if `.babel-fish/` exists):

```json
{
  "corpora": [
    {
      "name": "project-map",
      "root": "./.babel-fish",
      "glob": "**/*.md",
      "chunker": "markdown",
      "enabled": true,
      "extras": {
        "glossary_extractor": "babel-fish-vocabulary"
      }
    }
  ]
}
```

If smart-middle activation already detected `.babel-fish/01-vocabulary.md` on first run, this entry is auto-added.

### Verify

```bash
semantic-memory --notes . --stats
# Should show project-map corpus: <N> chunks, glossary.json: <N> entries

# Check the translation verbs are registered:
semantic-memory tools | grep -E "translate|reverse_translate|list_vocabulary"
```

## Workflow

### Authoring

1. Run babel-fish to generate / update the project map:
   ```bash
   babel-fish update
   ```
   (Or let the pre-commit hook do it automatically.)

2. Edit `.babel-fish/01-vocabulary.md` to add new vocabulary terms (see [`docs/glossary-contract.md`](./glossary-contract.md) for format).

3. semantic-memory's file-watcher detects the change and re-extracts `glossary.json` within seconds.

### Querying

In any Claude Code session with semantic-memory active:

```
"How does the deals page work?"
```

Claude can call:
```
translate("deals page")
→ { canonical_path: "features/deal-pipeline/DealPipeline.tsx", ... }

read_symbol("features/deal-pipeline/DealPipeline.tsx", "DealPipeline")
→ <code>
```

Without babel-fish + semantic-memory integration, Claude would scan files trying to figure out what "deals page" means. With it, the resolution is deterministic.

### Reverse lookup

```
"What's features/deal-pipeline/DealPipeline.tsx called in plain English?"
```

```
reverse_translate("features/deal-pipeline/DealPipeline.tsx")
→ [{ key: "deals page", section: "UI Pages", description: "..." }]
```

Useful when explaining LLM actions back to humans.

### Query rewriting (optional)

If you enable `vocabulary_query_rewrite: true` in the project-map corpus extras, semantic-memory preprocesses search queries:

```
search_code("how does the deals page work?")
↓ (auto-rewrite)
search_code("how does features/deal-pipeline/DealPipeline.tsx work?")
↓ (embed and search)
returns the actual code
```

This is **off by default** because silent rewriting can confuse users debugging searches. Enable it deliberately, knowing that's what's happening.

## Troubleshooting

### "translate verb is not registered"

Causes:
- semantic-memory < 1.1.0 (translation verbs are in 1.1.0+)
- `project-map` corpus not in `corpora.json`
- `.babel-fish/01-vocabulary.md` doesn't exist
- `glossary.json` extraction failed (check logs)

Verify:
```bash
ls .semantic/project-map/glossary.json   # should exist
cat .semantic/project-map/glossary.json | head  # should have entries
```

### "translate returns no results for a term I see in vocabulary.md"

Causes:
- Format mismatch — check the entry follows `- **<key>** → \`<path>\` — <desc>` exactly
- File-watcher hasn't detected the change yet — try `semantic-memory --notes . --reindex`
- The Unicode arrow `→` (U+2192) was replaced by `->` and the extractor's ASCII fallback isn't tolerant — re-add the Unicode arrow

### "Pre-commit hook regenerates the file but glossary.json doesn't update"

semantic-memory's file-watcher should catch it. If not:
- Check semantic-memory is actually running (`semantic-memory --notes . --stats`)
- Verify the watcher is active (`--no-watch` flag would disable it)
- Manual reindex: `semantic-memory --notes . --reindex`

### "Different project-maps in different projects — does babel-fish share state?"

No. Each project has its own `.babel-fish/` directory. Glossaries are per-project. There's no global vocabulary.
